## 1. Operator model (the "approval" mechanism)

There are NO allowances/amounts. An operator is time-bound and can move **any amount** on behalf of the holder.

```solidity
// storage: mapping(address holder => mapping(address spender => uint48)) private _operators;

function setOperator(address operator, uint48 until) public virtual;      // until = unix timestamp
function isOperator(address holder, address spender) public view virtual returns (bool) {
    return holder == spender || block.timestamp <= _operators[holder][spender];
}

function _setOperator(address holder, address operator, uint48 until) internal virtual; // emits OperatorSet

event OperatorSet(address indexed holder, address indexed operator, uint48 until);
```

Semantics:
- `holder == spender` is always an operator of itself.
- Approval is inclusive of the `until` timestamp (`block.timestamp <= until`).
- Revoke by calling `setOperator(operator, 0)` (or any past timestamp).
- Doc note in source: "Operators are given ACL allowance (ability to decrypt) for the transferred amount of transfers they initiate."

**For DisperseKit:** the user must first call `token.setOperator(disperseContract, until)` in a separate tx before the disperse tx.

## 2. Transfer functions — all overloads (ERC7984.sol v0.5.1, exact source)

All return the **actual transferred `euint64` handle** (which is encrypted-zero on insufficient balance, see §6 — NOT necessarily the requested amount).

```solidity
// externalEuint64 + inputProof variants (proof must be bound to the CALLING contract + user)
function confidentialTransfer(address to, externalEuint64 encryptedAmount, bytes calldata inputProof)
    public virtual returns (euint64);

function confidentialTransferFrom(address from, address to, externalEuint64 encryptedAmount, bytes calldata inputProof)
    public virtual returns (euint64);

// euint64-handle variants (caller must already have ACL access to `amount`)
function confidentialTransfer(address to, euint64 amount) public virtual returns (euint64);
function confidentialTransferFrom(address from, address to, euint64 amount) public virtual returns (euint64);

// transfer-and-call variants (receiver callback; see §7 — avoid for disperse)
function confidentialTransferAndCall(address to, externalEuint64 encryptedAmount, bytes calldata inputProof, bytes calldata data) public virtual returns (euint64);
function confidentialTransferAndCall(address to, euint64 amount, bytes calldata data) public virtual returns (euint64);
function confidentialTransferFromAndCall(address from, address to, externalEuint64 encryptedAmount, bytes calldata inputProof, bytes calldata data) public virtual returns (euint64);
function confidentialTransferFromAndCall(address from, address to, euint64 amount, bytes calldata data) public virtual returns (euint64);
```

Exact checks inside each (from source):

```solidity
function confidentialTransfer(address to, euint64 amount) public virtual returns (euint64) {
    require(FHE.isAllowed(amount, msg.sender), ERC7984UnauthorizedUseOfEncryptedAmount(amount, msg.sender));
    return _transfer(msg.sender, to, amount);
}

function confidentialTransferFrom(address from, address to, euint64 amount) public virtual returns (euint64) {
    require(FHE.isAllowed(amount, msg.sender), ERC7984UnauthorizedUseOfEncryptedAmount(amount, msg.sender));
    require(isOperator(from, msg.sender), ERC7984UnauthorizedSpender(from, msg.sender));
    euint64 transferred = _transfer(from, to, amount);
    FHE.allowTransient(transferred, msg.sender);   // caller can use the returned handle in same tx
    return transferred;
}

function confidentialTransferFrom(address from, address to, externalEuint64 encryptedAmount, bytes calldata inputProof)
    public virtual returns (euint64) {
    require(isOperator(from, msg.sender), ERC7984UnauthorizedSpender(from, msg.sender));
    euint64 transferred = _transfer(from, to, FHE.fromExternal(encryptedAmount, inputProof));
    FHE.allowTransient(transferred, msg.sender);
    return transferred;
}
```

Notes:
- Token uses `FHE.isAllowed(amount, msg.sender)` — NOT `FHE.isSenderAllowed` — but the effect is the same: the direct caller must have (persistent or transient) ACL access to the amount handle.
- Both `confidentialTransferFrom` overloads grant `FHE.allowTransient(transferred, msg.sender)` on the return value, so a disperse contract can do FHE math on what was actually moved within the same tx.
- Plain `confidentialTransfer` (non-From) does not add a transient grant for the caller, but `_update` already does `FHE.allow(transferred, from)` and `from == msg.sender`, so the caller is persistently allowed anyway.
- The `externalEuint64` variant of `confidentialTransferFrom` would require the input proof to be bound to the **token** contract address (since the token calls `FHE.fromExternal`) — impractical for a disperse contract that needs to split one encrypted total; use the `euint64` overload.

### Event

Emitted from `_update` on EVERY transfer/mint/burn, with the encrypted **actually-transferred** amount handle (all three params indexed):

```solidity
event ConfidentialTransfer(address indexed from, address indexed to, euint64 indexed amount);
```

`from == address(0)` → mint; `to == address(0)` → burn. The `amount` is the euint64 handle (bytes32 under the hood) — observers learn only the handle, not the value. On a failed (insufficient-balance) transfer the event still fires with a handle that decrypts to 0.

Also available: `event AmountDisclosed(euint64 indexed encryptedAmount, uint64 amount)` + `requestDiscloseEncryptedAmount(euint64)` / `discloseEncryptedAmount(euint64, uint64, bytes)` for voluntary public disclosure of an amount, and `event AmountDiscloseRequested(euint64 indexed encryptedAmount, address indexed requester)`.

## 3. Balances and ACL on balance handles

```solidity
function confidentialBalanceOf(address account) public view virtual returns (euint64);
function confidentialTotalSupply() public view virtual returns (euint64);
```

ACL on each new balance handle, set in `_update`:

```solidity
FHE.allowThis(ptr);        // the token contract itself
FHE.allow(ptr, from);      // the holder (sender side)
...
FHE.allow(ptr, to);        // the holder (receiver side)
```

So **only the token contract and the account holder** can use/decrypt a balance handle. The disperse contract, operators, and third parties CANNOT decrypt balances. (The frontend decrypts the user's balance via the relayer SDK `userDecrypt` with the user's signature.) The `ERC7984ObserverAccess` extension exists if per-account observers are needed — not needed for DisperseKit.

ACL on the transferred amount handle, set in `_update`:

```solidity
if (from != address(0)) FHE.allow(transferred, from);
if (to != address(0)) FHE.allow(transferred, to);
FHE.allowThis(transferred);
```

i.e. sender, recipient, and token get persistent access to each transfer's amount handle; the operator (disperse contract) additionally gets a transient grant (§2).

## 4. Minting and a minimal demo token

Internal API (ERC7984.sol):

```solidity
function _mint(address to, euint64 amount) internal returns (euint64 transferred);   // reverts on to == address(0)
function _burn(address from, euint64 amount) internal returns (euint64 transferred);
function _transfer(address from, address to, euint64 amount) internal returns (euint64 transferred);
function _update(address from, address to, euint64 amount) internal virtual returns (euint64 transferred);
```

`_mint` uses `FHESafeMath.tryIncrease` on total supply — if supply would overflow uint64, the mint silently mints an encrypted 0 (no revert).

Constructor: `constructor(string memory name_, string memory symbol_, string memory contractURI_)`.

**Important:** `ERC7984` is abstract and does **NOT** set the Zama coprocessor config. The concrete token must inherit the config contract. In `@fhevm/solidity` **0.11.1** the config contract is **`ZamaEthereumConfig`** (from `@fhevm/solidity/config/ZamaConfig.sol`); it selects addresses by chainid (1 = mainnet, 11155111 = Sepolia, 31337 = local hardhat) and reverts `ZamaProtocolUnsupported` elsewhere. The `SepoliaConfig` contract name from older `@fhevm/solidity` versions/docs no longer exists in 0.11.1 — only a private `_getSepoliaConfig()` helper remains.

Sepolia protocol addresses baked into ZamaConfig 0.11.1:
- ACL: `0xf0Ffdc93b7E186bC2f8CB3dAA75D86d1930A433D`
- Coprocessor (FHEVMExecutor): `0x92C920834Ec8941d2C77D188936E1f7A6f49c127`
- KMSVerifier: `0xbE0E383937d564D7FF0BC3b46c51f0bF8d5C311A`

### Complete minimal demo token with public faucet

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.27;

import {FHE, euint64, externalEuint64} from "@fhevm/solidity/lib/FHE.sol";
import {ZamaEthereumConfig} from "@fhevm/solidity/config/ZamaConfig.sol";
import {ERC7984} from "@openzeppelin/confidential-contracts/token/ERC7984/ERC7984.sol";

/// @dev Demo confidential token: anyone can mint themselves a fixed clear amount.
contract DemoConfidentialToken is ERC7984, ZamaEthereumConfig {
    uint64 public constant FAUCET_AMOUNT = 1_000_000_000; // 1000 tokens @ 6 decimals

    constructor() ERC7984("Demo Confidential USD", "dcUSD", "") {}

    /// Public faucet: mints a trivially-encrypted (publicly known) amount to the caller.
    function mint() external {
        _mint(msg.sender, FHE.asEuint64(FAUCET_AMOUNT));
        // _update handles all ACL: FHE.allowThis + FHE.allow(balance, msg.sender)
    }

    /// Optional: mint an encrypted, caller-chosen amount (proof bound to THIS token + user).
    function mint(externalEuint64 encryptedAmount, bytes calldata inputProof) external {
        _mint(msg.sender, FHE.fromExternal(encryptedAmount, inputProof));
    }
}
```

Notes: `FHE.asEuint64(uint64)` is a trivial encryption (value is public on-chain — fine for a faucet). No extra ACL calls needed around `_mint`; `_update` handles handle permissions. This compiles with solc >= 0.8.27 (union of pragmas: ERC7984 `^0.8.27`, FHE `^0.8.24`).

## 5. ACL requirements for a third-party disperse contract

Flow: frontend encrypts per-recipient amounts with `createEncryptedInput(disperseContractAddress, userAddress)` → disperse contract does `FHE.fromExternal(extAmt, proof)` → calls `token.confidentialTransferFrom(user, recipient, euint64Amt)`.

What each step grants/checks (verified in source):

1. `FHE.fromExternal` (Impl.verify, Impl.sol L670-674): calls `FHEVMExecutor.verifyInput(inputHandle, msg.sender, inputProof, type)` then `ACL.allowTransient(result, msg.sender)`. The resulting `euint64` is **transiently allowed to the disperse contract only**. The input proof is bound to (contract = disperse contract, user = tx signer) — the frontend must build it for the disperse contract's address, not the token's.

2. `ERC7984.confidentialTransferFrom(from, to, euint64 amount)` requires:
   - `FHE.isAllowed(amount, msg.sender)` — msg.sender is the disperse contract; satisfied by the transient allowance from step 1. (The token uses `FHE.isAllowed(amount, msg.sender)`, functionally equivalent to `FHE.isSenderAllowed(amount)` here.)
   - `isOperator(from, msg.sender)` — user must have called `setOperator(disperse, until)` earlier.

3. **YES, `FHE.allowTransient(amount, address(token))` is required before the call.** The token itself performs FHE operations on `amount` inside `_update`/`FHESafeMath` (`FHE.ge`, `FHE.select`, `FHE.sub`, `FHE.add`). The FHEVMExecutor enforces that the contract invoking each FHE op has ACL access to its operand handles, and the transient grant from `fromExternal` covers only the disperse contract. Without granting the token access, the token's own FHE ops on the handle revert at the coprocessor ACL check. So the disperse loop body is:

```solidity
euint64 amt = FHE.fromExternal(extAmounts[i], inputProof);  // one proof can carry many inputs
FHE.allowTransient(amt, address(token));                    // let the TOKEN compute on the handle
euint64 sent = token.confidentialTransferFrom(msg.sender, recipients[i], amt);
// `sent` is transiently allowed to this contract (granted inside confidentialTransferFrom)
```

   (This matches the pattern in Zama's official docs for contracts composing over confidential tokens. The exact coprocessor-side revert behavior is enforced by the deployed FHEVMExecutor/ACL, not visible in library source — the requirement itself is standard Zama guidance. Marked CONFIRMED for the grant pattern, the token-side `FHE.isAllowed` gate, and the `allowTransient` semantics in `Impl.sol`/`IACL`; the executor-side operand check is from Zama protocol docs, not re-derived here.)

4. `ACL.allowTransient` note (Impl.sol IACL docs): "The caller must be allowed to use handle for allowTransient() to succeed. If not, allowTransient() reverts." The disperse contract IS allowed (step 1), so it can grant to the token. Transient allowances last only for the transaction.

## 6. Insufficient balance = encrypted zero, no revert (CONFIRMED from source)

`_update` (ERC7984.sol L290-323):

```solidity
(success, ptr) = FHESafeMath.tryDecrease(fromBalance, amount);  // success = FHE.ge(oldValue, delta)
...
transferred = FHE.select(success, amount, FHE.asEuint64(0));
```

If the sender's encrypted balance < amount, the balance is left unchanged and **an encrypted 0 is "transferred"** — the tx succeeds, the `ConfidentialTransfer` event fires, and the returned handle decrypts to 0. A disperse contract therefore cannot detect failure synchronously; the returned `transferred` handles (transiently allowed to the caller) are the only signal, and reading them requires decryption (or `FHE.makePubliclyDecryptable` / callback flows). Same applies to mint overflow (`tryIncrease`).

Uninitialized balances: an account that never received tokens has `euint64.wrap(0)`; `FHESafeMath` treats uninitialized as 0 explicitly.

## 7. Receiver/callback variants — avoid for disperse

- `confidentialTransferAndCall` / `confidentialTransferFromAndCall` invoke `IERC7984Receiver(to).onConfidentialTransferReceived(operator, from, amount, data) returns (ebool)` on contract recipients (EOAs skip the callback) and attempt a best-effort refund if it returns encrypted-false. Source warns refunds are best-effort and can fail silently.
- Plain `confidentialTransfer` / `confidentialTransferFrom` NEVER call the receiver — contract recipients receive tokens unconditionally, like ERC-20. **Use the plain `confidentialTransferFrom(address,address,euint64)` overload for dispersing.**
- If DisperseKit's own contract might ever RECEIVE ...AndCall transfers, it would need to implement `IERC7984Receiver`; not needed otherwise.

```solidity
interface IERC7984Receiver {
    function onConfidentialTransferReceived(address operator, address from, euint64 amount, bytes calldata data)
        external returns (ebool);
}
```

## 8. Other facts / gotchas

- Errors: `ERC7984InvalidReceiver(address)`, `ERC7984InvalidSender(address)`, `ERC7984UnauthorizedSpender(address holder, address spender)`, `ERC7984UnauthorizedUseOfEncryptedAmount(euint64 amount, address user)`, `ERC7984UnauthorizedCaller(address)`.
- `supportsInterface(type(IERC7984).interfaceId)` is implemented (ERC-165).
- One input proof can bundle multiple `externalEuint64` handles (relayer SDK `createEncryptedInput` supports multiple `add64` calls → one proof) — ideal for a disperse call taking `externalEuint64[] amounts, bytes inputProof`.
- Extensions available in v0.5.1 (not needed for MVP): `ERC7984ERC20Wrapper` (wrap an ERC-20 into confidential form — potentially interesting for demoing with existing tokens), `ERC7984Freezable`, `ERC7984Restricted`, `ERC7984Rwa`, `ERC7984Votes`, `ERC7984Omnibus`, `ERC7984ObserverAccess`, `ERC7984Hooked`.
- UNVERIFIED (out of scope of package source): exact relayer SDK call shapes (`@zama-fhe/relayer-sdk`) and Sepolia relayer URL; verify in a separate SDK research pass.
